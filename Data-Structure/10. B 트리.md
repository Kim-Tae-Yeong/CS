# 10. B 트리
## 10.1 메모리 계층 구조
컴퓨터에는 많은 종류의 메모리가 있다.
일반적으로 생각하는 메인 메모리가 있고, CPU에는 레지스터라는 메모리가 있다.
CPU와 메인 메모리 사이에는 캐시가 있다.
그리고 하드 디스크가 있다.
이 메모리들은 계층 구조를 이루고 있다.

그림으로 살펴보자.

이 그림에서 위로 올라갈수록 속도는 빠르지만 용량은 적다.
그리고 아래로 내려갈수록 속도는 느리지만 용량은 커진다.

오른쪽에는 CPU가 데이터를 요청했을 때 해당 메모리에서 데이터를 가져오는 클럭 수를 표현했다.
메인 메모리만 해도 20클럭으로 느리지만, 하드 디스크 속도와 비교할 것은 안된다.
하드 디스크에서 데이터를 가져오는 것은 그만큼 리소스가 많이 필요한 연산이다.
물론 최근에는 SSD 등 성능이 향상된 보조 기억 장치가 등장하여 그림과 같이 느리지는 않겠지만, 그래도 속도가 느린 것은 여전하다.

대부분의 자료 구조는 메인 메모리를 기준으로 설계했다.
어찌 보면 당연한 것이다.
프로세스를 시작하면 하드 디스크에서 필요한 메모리를 메인 메모리로 가져오고, 이 메모리를 기반으로 프로그램을 수행하기 때문이다.
프로그래밍을 처음 공부할 때 "데이터베이스에 접근하는 횟수를 최대한 줄여라"라는 말을 많이 듣는다.
데이터베이스는 데이터를 하드 디스크에 저장하기 때문이다.
데이터베이스는 하드 디스크에 저장되는 데이터를 최대한 효율적으로 관리하는 것 뿐이지 하드 디스크가 가지는 하드웨어적 한계를 뛰어넘는 것은 아니다.

그렇다면 데이터베이스는 어떻게 데이터를 삽입, 탐색, 삭제할까?
데이터베이스가 사용하는 자료 구조를 공부하면 앞으로 데이터베이스를 설계하고 SQL을 작성하는 데 도움을 많이 받을 수 있다.
테이블에 레코드를 저장하고, 읽어 오고, 삭제하는 데 사용하는 자료 구조가 바로 B 트리와 B 트리의 변형인 B+ 트리이다.
많은 데이터베이스가 B+ 트리를 사용한다.

다음 절에서는 B 트리를 알아보기 전에 먼저 B 트리를 사용하는 데이터베이스에서 데이터가 삽입, 탐색, 삭제되는 과정을 훑어보자.
데이터베이스를 이해해야 B 트리가 내부적으로 어떻게 작동하는지 이해하기 쉬울 것이다.

## 10.3 B 트리
B 트리는 m - way 탐색 트리(m - way search tree)의 일종이다.
여기서 m은 노드가 가질 수 있는 최대 자식 개수를 의미한다.
BST의 노드는 자식을 두 개 가질 수 있다.
그러므로 BST는 2 - way 탐색 트리이다.
자식은 둘이지만 노드가 가지는 키는 하나이다.
이를 일반화하면 m - 1은 노드가 가질 수 있는 최대 키 개수이다.

m - way 탐색 트리에서 중요한 점은 노드에 키가 여러 개 있을 수 있다는 것이다.
여기서 한 가지 궁금점이 생긴다.
노드에 키가 하나일 때와 여러 개일 때 어떤 차이가 있을까?
어떤 상황에서 노드에 키를 여러 개 두어야 할까?
이진 탐색 트리가 결국에는 선형 탐색을 피하고자 트리에 이진 탐색을 적용한 것인데, m을 1001처럼 큰 수로 잡는다면 다시 선형 탐색으로 회귀하는 모습이다.
노드에 있는 키 개수 차이는 메모리 접근 횟수와 비교 연산 횟수의 차이를 의미한다.

그림으로 확인해보자.

<img width="445" alt="image" src="https://github.com/user-attachments/assets/1cf86eb4-9365-463d-bb31-31e6bea990f1" />

그림 10 - 2를 보면 왼쪽이 BST고 오른쪽이 4 - way 탐색 트리이다.
찾고자 하는 키는 15이다.
두 자료 구조 모두 하드 디스크에 저장되어 있다고 해보자.
BST에서 노드는 하나의 키를 가지고 있기 때문에 하드 디스크에 저장된 노드를 읽어 오는 메모리 접근 횟수, 읽어온 데이터를 찾고자 하는 키와 비교하는 연산 횟수 모두 트리 높이(height of tree)와 같다.

BST가 완전 이진 트리라면 트리 높이 h = $\log n$이 된다.
오른쪽의 4 - way 탐색 트리를 살펴보면 m이 2에서 4로 커지면서 트리 높이는 2로 줄어들었다.
하드 디스크에서 메인 메모리로 노드를 하나씩 가져온다고 했을 때 트리 높이가 줄어들었다는 이야기는 메모리 접근 횟수가 줄어들었다는 것을 의미한다.
실제로 하드 디스크에서 메인 메모리로 가져올 때는 큰 단위로 가져오는 데. 가상 메모리가 적용된 시스템은 페이지(page) 크기(일반적으로 4KB = 4096byte)만큼 가져온다.
그 대신 한 노드에서 찾고하 하는 키를 찾기 위해 선형 탐색을 하기 때문에 비교 연산 횟수는 늘어난다.

메모리 접근 횟수는 줄어든 대신 비교 연산 횟수는 늘어났으니 결국 성능은 비슷하지 않나 하는 의문이 들 수 있지만 그렇지 않다.
이 자료 구조가 하드 디스크에 저장되어 있기 때문이다.
비교 연산은 메인 메모리에 있는 두 값을 가져다 뺄셈 한 번만 하면(비교 연산은 CPU에서 두 수를 빼서 알아냄) 알 수 있다.
이때 두 값을 가져오는 데 40클럭이 걸린다.
여기에 연산 후 값을 메모리에 쓰는데 20클럭 그리고 뺄셈 연산에 1클럭을 더해 61클럭이면 충분하지만, 메모리 접근 횟수는 하드 디스크에 대한 접근이므로 한 번에 50만 클럭이 필요하다.
자료 구조가 어디에 저장되어 있느냐에 따라 메모리 접근과 비교 연산의 차이가 이렇게도 극명하게 난다.

결론은 나왔다.
하드 디스크에 상주하는 자료 구조를 설계한다면 한 노드에 최대한 많은 키를 담아 메모리 접근 횟수를 줄여야 한다.
그렇다고 노드 크기를 무작정 늘릴 수도 없다.
그렇게 되면 노드 크기가 페이지 크기를 넘게 되어 한 번에 한 노드를 가져올 수 없기 때문이다.
그래서 실제로는 페이지 크기에 맞추어서 m 값을 수백 정도로 설정한다.

B 트리는 m - way 탐색 트리의 일종이다.
먼저 m - way 탐색 트리를 정의해보자.

1. 서브 트리를 최대 m개 가진다.
2. 한 노드에서 key는 정렬되어 있다.
3. 어떤 키 K의 왼쪽 서브 트리의 모든 키는 K보다 작다.
4. 어떤 키 K의 오른쪽 서브 트리의 모든 키는 K보다 크다.
5. 서브 트리도 m - way 탐색 트리이다.

세 번째와 네 번째 특징은 마치 BST와 유사하다.
탐색 트리는 모두 BST를 기반으로 하기 때문이다.
그림 10 - 2와 비교하면서 읽어 보면 훨씬 쉽게 이해할 수 있다.
여기에 더해 B 트리의 특징을 알아보자.
차수가 m인 B 트리는 아래와 같이 정의할 수 있다.

1. 2 <= 루트 노드의 서브 트리 개수 <= m
2. ⎡m / 2⎤ <= 루트를 제외한 노드의 서브 트리 개수 <= m
3. ⎡m / 2⎤ - 1 <= 루트를 제외한 노드의 키 개수 <= m - 1
4. 모든 리프 노드는 같은 레벨에 있다.

키 앞에 자식을 가리키는 참조 하나 뒤에 참조 하나가 있다.

차수 m이 3인 트리를 따로 2 - 3 트리(2 - 3 tree)라고도 하는데, 다음 절에서 살펴볼 B 트리에서의 삽입과 삭제 연산에는 2 - 3 트리를 사용하겠다.

## 10.4 B 트리에 키 삽입 / 삭제하기

그렇게 계속 내려가다 보면 찾고자 하는 키를 찾을 수 있을 것이다.

## 10.5 B+ 트리
B+ 트리는 B 트리의 변형으로 실제 데이터베이스는 B+ 트리로 구현된 경우가 많다.
B+ 트리의 삽입, 탐색, 삭제는 B 트리와 매우 유사하므로, 이 절에서는 단순히 B+ 트리가 B 트리와 무엇이 다른지 그 구조만 살펴보자.

B+ 트리를 그림으로 먼저 살펴보자.

<img width="367" alt="image" src="https://github.com/user-attachments/assets/8e373e9b-1530-4d5b-960e-79c18a4aa306" />

그림 10 - 19는 B+ 트리이다.
B 트리와 거의 같다.
다른 점만 정리해보자.

1. 노드는 인덱스 노드(index node)와 리프 노드인 데이터 노드(data node)로 구성된다. B 트리는 모든 노드 안에 있는 키 바로 옆에 테이블의 레코드를 참조하는 레코드 포인터가 있다. 하지만 B+ 트리에서 인덱스 노드에는 레코드 포인터가 없고 다른 인덱스 노드나 데이터 노드에 대한 참조만 있다.
2. 모든 레코드 포인터는 데이터 노드에 있다. 실제 데이터를 가리키고 있기 때문에 데이터 노드이다. 가장 중요한 점은 데이터 노드에 모든 키가 다 있다는 것이다. B 트리와 비교해보자. 루트 노드에 있는 키 5도 세 번째 데이터 노드에 있고, 키 3도 키 3이 있는 노드의 오른쪽 자식 노드에 있다. 키 7도 키 7이 있는 노드의 오른쪽 자식 노드에 있다. 이렇게 인덱스 노드에 있는 모든 키는 데이터 노드에 있다. 하지만 인덱스 노드에서는 키의 중복이 없다. 어떤 규칙에 따라 인덱스 노드의 키들이 데이터 노드에서 중복해서 나타날까? 조금만 살펴보면 그 키의 오른쪽 서브 트리에서 가장 작은 키가 있어야 할 곳에 해당 키가 중복되어 나타난다. 루트 노드에 있는 키 5를 보면 오른쪽 자식 노드인 키 7을 가진 노드의 왼쪽 자식 노드의 첫 번째 키가 5이다. 오른쪽 서브 트리에서 가장 작은 키가 들어갈 위치이다. 모든 인덱스 노드의 키들도 이러한 규칙에 따라 데이터 노드에 중복하여 나타난다.
3. 데이터 노드들은 이중 연결 리스트로 연결되어 있다. 그림 10 - 19에서도 화살표를 통해 모든 데이터 노드가 연결 리스트로 연결된 모습을 볼 수 있다.

이로써 B 트리와 B+ 트리를 모두 알아보았다.
이전 절에서 테이블에서 PRIMARY KEY에 대해 B 트리로 만들어진 인덱스가 자동 생성되는 것을 살펴보았다.
이번에는 다른 컬럼에 대한 인덱스를 만들어보자.

## 10.6 B 트리로 인덱스 만들기
데이터베이스에서 테이블을 만들면 자동으로 B+ 트리로 구성된 인덱스가 생성된다.
SELECT 구문에서 WHERE 절은 B 트리에서 키를 탐색하는데 사용한다.

코드 10 - 7은 example 테이블에서 ID가 2인 데이터를 탐색하는 SQL이다.

```
코드 10 - 7
MariaDB [mydb]> SELECT * FROM exmaple
  -> WHERE ID = 2;
```

<img width="130" alt="image" src="https://github.com/user-attachments/assets/1264625c-832c-4654-bc20-552f1def4ac2" />

코드 10 - 7을 보면 SELECT 문에 WHERE 절이 보이는데, 여기에 조건으로 있는 ID가 B 트리의 키이다.
레코드가 꽤 많아도 B 트리가 가지는 장점들 때문에 굉장히 빠른 속도로 탐색 가능하다.
인덱스가 없었다면 선형 탐색을 하게 되어 굉장히 오랜 시간이 걸릴 것이다.

그런데 갑자기 이름을 탐색해야 하는 상황이 왔다고 해보자.
레코드는 1만 개 정도 된다고 해보자.
그렇다면 모든 레코드에서 선형 탐색을 해야 하니 O(n)이 걸릴 것이다.
게다가 레코드를 하드 디스크에서 가져와야 한다.
이름에서도 B 트리를 만들어 검색 속도를 향상시킬 수 없을까?
가능하다.
이름에 대해서 인덱스를 생성하면 된다.
그럼 이름을 키로 B 트리를 구성한다.

코드 10 - 8은 인덱스를 생성하는 SQL이다.

```
코드 10 - 8
MariaDB [mydb]> CREATE INDEX idx_name
  -> ON example(name);
```

인덱스 idx_name을 테이블 example의 컬럼 name에 대해 생성한 것이다.
이제 B 트리가 생성되었으니 탐색 속도는 놀랄 만큼 향상될 것이다.

코드 10 - 9는 이름이 '이순신'인 데이터를 탐색하는 SQL이다.

```
코드 10 - 9
MariaDB [mydb]> SELECT * FROM example
  -> WHERE name LIKE '이순신';
```

<img width="130" alt="image" src="https://github.com/user-attachments/assets/56187070-b1c6-43e3-9847-17e9e88ddf11" />

코드 10 - 9를 보면 WHERE 절에서 name을 사용하고 있다.
name에 대해 인덱스를 생성해 두었으므로 B+ 트리를 사용해서 탐색할 것이다.

인덱스를 생성할 때는 유의 사항이 있다.
탐색 속도를 향상시킬 목적으로 생성한 인덱스를 더 이상 탐색할 필요가 없을 때는 반드시 삭제하는 것이 좋다.
인덱스는 B+ 트리를 구성한다.
이 때문에 탐색 속도는 월등히 향상되지만, 레코드의 삽입과 삭제가 빈번히 일어나면 insert 연산의 split이나 delete 연산의 donate, merge 등이 발생하여 데이터베이스 성능을 크게 저하시킬 수 있다.
게다가 테이블 크기가 커지면 B+ 트리 자체의 크기도 커져 용량을 차지한다.
물론 B+ 트리도 하드 디스크에 저장되므로 용량을 크게 신경 쓰지 않아도 된다고 생각할 지 모르지만, 리소스가 낭비되는 것은 엄연한 사실이다.
그러므로 인덱스를 더 이상 사용하지 않을 때는 반드시 삭제해야 한다.
