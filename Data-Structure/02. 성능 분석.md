# 2. 성능 분석
## 2.1 자료 구조 성능 이야기 : 빅오
자료 구조를 공부하는 중요한 이유 중 하나는 어떤 상황에서 어떤 자료 구조를 사용해야 좀 더 효율적으로, 다시 말해 성능이 좋은 프로그램을 만들 수 있는지 알기 위해서이다.
그렇다면 먼저 알고리즘 성능을 어떻게 분석해야 하는지부터 알아야한다.

### 2.1.1 알고리즘 성능 분석
어떤 자료 구조의 연산(일반적으로 함수로 구현되는 연산)이 다른 자료 구조의 특정 연산에 비해 성능이 좋다고 이야기하려면 두 연산의 성능을 비교할 수 있어야한다.
그렇다면 성능은 어떻게 비교할 수 있을까?
간단히 두 가지 정도의 방법이 떠오른다.

첫 번째 방법은 절대 시간을 재는 것이다.
하지만 이 방법은 현실적이지 않다.
두 함수를 비교하려면 같은 환경의 시스템에서 테스트해야 한다는 제약 조건이 있기 때문이다.
두 함수의 성능을 비교하면서 한 함수의 성능은 A 컴퓨터에서 2초이고 다른 함수의 성능은 B 컴퓨터에서 5초라고 한다면, 첫 번째 함수가 두 번째 함수보다 성능이 좋은 것일까?
B 컴퓨터의 성능이 A 컴퓨터의 성능보다 월등히 좋다면 머릿속은 더 복잡해진다.
이래서는 의미가 없다.

두 번째 방법은 어떤 기준을 설정하고 이에 맞추어 두 함수를 상대적으로 비교하는 것이다.
예를 들어 함수 내에 있는 반복문이나 분기문의 실행 횟수, 덧셈이나 곱셈 같은 연산 횟수를 세는 것이다.
이렇게 하면 실행 환경과는 분리해서 독립적으로 성능을 분석할 수 있다.
그럼 간단한 예로 연산 횟수를 세어보자.

```
코드 2 - 1 operation_count.py
def func(arr, init):  # 연산  빈도수
  n = len(arr)        #   1     1
  ret = init          #   1     1
  for i in range(n):  #   1     n
    ret += arr[i]     #   1     n
  return ret          #   1     1
                      #    2n + 3

  arr = [1, 2, 3, 4, 5]
  result = func(arr, 10)
  print(result)
```

이 코드를 실행하면 25라는 결과값이 출력된다.

코드 2 - 1에서 성능을 판단하고 싶은 함수는 func이다.
매개변수 arr은 리스트이며 init은 초깃값이다.
단순히 초깃값에 arr의 모든 요소 값을 더한 결괏값을 구하는 함수이다.
주석에서 func 함수의 연산 횟수를 계산하는 과정을 기술했다.
살펴보면 n = len(arr)은 arr 길이를 n에 할당하는 연산이다.
이 연산은 함수를 통틀어 한번만 계산된다.
ret = init도 마찬가지이다.
우리가 유의해서 보아야 할 곳은 for문이다.
range()에서 i 값을 반환받는데, 이 연산은 arr 길이만큼 반복된다.
즉, 빈도수가 n이 되어 함수가 실행될 때 n번 실행된다.
ret += arr[i]도 마찬가지이다.
최종적으로 얻은 함수의 연산 횟수는 arr 길이 n에 대한 함수 2n + 3이 된다.

$T(n) = 2n + 3$

이 예제는 아주 간단하므로 연산 횟수 $T(n)$을 쉽게 구할 수 있었지만, 일반적인 경우 연산 횟수를 정확하게 기술하기란 쉽지 않다.
또 아주 애매한 상황과 마주칠 수 있는데, 다른 함수의 연산 횟수 $T(n) = n + 500$이라면 이 함수는 코드 2 - 1의 함수 func보다 성능이 좋을까 나쁠까?
판단하는 사람에 따라 1차항의 계수가 큰 func 함수에서 arr 길이가 길어질수록 2배만큼 더 연산을 해야 하니 다른 함수가 성능이 좋다고 말할 수도 있다.
만약 $T(n) = 2n + 5$라면 func 함수가 이 함수에 비해 반드시 성능이 좋다고 말할 수 있을까?

이런 모호함을 없애고자 알고리즘 성능을 분석할 때는 점근적 표기법을 사용한다.
많은 사람이 빅오(big - O)로 알고 있는 이 점근적 표기법에는 사실 O - 표기법뿐만 아니라 θ - 표기법과 Ω - 표기법도 있다.
이 세 가지 점근적 표기법을 알아보자.

### 2.1.2 성능을 비교하는 방법 : 빅오
먼저 빅오 정의를 살펴보면 다음과 같다.

> 모든 $n > n_0$에 대해 $O <= f(n) <= cg(n)$인 양의 상수 $c, n_0$가 존재하면 $f(n) = O(g(n))$

정의만으로는 어떤 의미인지 알기 어렵다.
코드 2 - 1의 예로 빅오 정의를 이해해보자.
$T(n) = 2n + 3$은 $n$에 대한 선형 함수이다.
선형 함수의 특징은 기울기가 증가할수록 그래프는 가파르게 그려진다는 것이다.

그림으로 확인해보자.

<img width="255" alt="image" src="https://github.com/user-attachments/assets/a7779a9d-8c7a-45e7-b9f3-ae52fe8318f3" />

그림 2 - 1에서 확인할 수 있듯이 기울기가 4인 함수가 1인 함수보다 가파르게 그려진다.
이를 이용하면 정의에서 나온 조건인 상수 $n_0$와 $c$를 구할 수 있다.
$T(n) = 2n + 3$을 $f(n)$이라고 하고 $g(n) = n$이라고 하자.
그럼 $n_0$와 $c$는 굉장히 많은 쌍을 구할 수 있다.
그림 중 한가지 예로 $n_0 = 3, c = 3$으로 특정해 보면 그래프는 다음 그림과 같이 그려진다.

<img width="249" alt="image" src="https://github.com/user-attachments/assets/dd286c2c-4119-4081-aff5-9e2086590ef5" />

$cg(n) = 3n$이 되고 $cg(n)$과 $f(n)$이 만나는 지점인 $n_0$ 이후로는 항상 모든 $n$에 대해 $cg(n) >= f(n)$이다.
$f(n) > 0$도 항상 만족한다.
이때는 $2n + 3 = O(n)$이라고 표기할 수 있는데, $g(n)$이 $n$이었고 $f(n)$이 $2n + 3$이었기 때문이다.
빅오는 상한(upper bound)을 의미한다.
$2n + 3 = O(n)$은 $f(n) = 2n + 3$이 $n_0$ 이상에서 그래프의 증가 추세가 절대 $cg(n) = cn$을 넘지 않는다는 것을 보여준다.
그래프를 보면 당연한 이야기이다.

이 빅오 표기법을 읽을 때 두 가지 주의해야 할 사항이 있다.
첫 번째로 $2n + 3 = O(n)$에서 =를 등호로 해석해서는 안된다.
그 대신 " $2n + 3$의 빅오는 $n$이다."라고 읽어야 한다.
두 번째로 조금만 생각해 보면 $2n + 3 = O(n ^ 2)$이란 표현도 가능하다는 것을 알 수 있다.
2차 함수인 $n ^ 2$은 어느 시점에서 반드시 $2n + 3$보다 클 것이기 때문이다.
이 표기가 틀린 것은 아니지만 이래서는 별로 의미가 없다.
우리가 알고 싶은 것은 성능의 상한이다.
한계를 알고 싶은데 차이가 급격하게 나는 비교 대상을 언급하는 것은 아무런 도움도 되지 않는다.

점근적 표기법에는 O - 표기법 외에도 두 가지가 더 있다.
θ - 표기법과 Ω - 표기법이다.
이 두 가지 표기법은 잘 사용하지 않기 때문에 간단하게 짚고 넘어가자.

먼저 θ - 표기법 정의는 아래과 같다.

> 모든 $n > n_0$에 대해 $c_1g(n) <= f(n) <= c_2g(n)$인 양의 상수 $c_1, c_2, n_0$가 존재하면 $f(n) = θg(n))$

O - 표기법과 유사한데 $c_1g(n) <= f(n)$이 추가된 형태이다.
O - 표기법처럼 $f(n) <= c_2g(n)$은 상한을 의미한다.
이와 유사하게 $c_1g(n) <= f(n)$은 하한(lower bound)을 의미한다.
예를 들어 쉽게 접근해보자.
어떤 함수가 $θ(n)$이라면 이 함수의 성능은 데이터 개수 $n$이 늘어날 때 절대 선형 시간보다 성능이 좋을 수 없다.
(선형 시간보다 좋은 성능으로는 $\log n$이나 상수 시간이 있다. 대표적인 빅오는 잠시 후에 살펴보자.)
θ - 표기법은 O - 표기법보다 더 많은 정보를 제공한다.
상한과 함께 하한을 제공하기 때문이다.
구할 수 있다면 θ - 표기법을 구하는 것이 좋겠지만 빅오를 구하는 것도 벅찬 상황에서 이를 구하기란 매우 어려워 실제로는 O - 표기법을 주로 사용한다.

Ω - 표기법 정의는 아래와 같다.

> 모든 $n > n_0$에 대해 $cg(n) <= f(n)$인 양의 상수 $c, n_0$가 존재하면 $f(n) = Ω(g(n))$

Ω - 표기법은 하한을 의미한다.
실제로는 거의 사용하지 않는다.

마지막으로 대표적인 빅오 종류를 성능이 빠른 순서대로 쭉 훑어보자.

1. $O(1)$ : 상수 시간이라고 한다. 자료 구조에 저장된 데이터 개수와 상관없이 정해진 횟수의 연산 안에 알고리즘이 마무리된다. 가장 좋은 예로 동적 배열의 인덱싱이 있다. 동적 배열은 다음 장에서 알아보자.
2. $O(\log n)$ : 로그 시간이라고 한다. 로그의 그래프를 생각해 보면 이는 $y = x$로 그려지는 선형 시간에 비해 성능이 월등히 좋다고 할 수 있다. 대표적인 예로 이진 탐색 트리 계열에 속하는 자료 구조(이진 탐색 트리, 균형 이진 트리, B 트리)의 삽입과 탐색, 삭제 모두 로그 시간이다.
3. $O(n)$ : 선형 시간이다. 데이터 개수가 늘어날수록 선형으로 연산 횟수가 늘어난다. 실제 함수를 만드는 데 $O(n)$으로 만들 수 있다면 충분히 만족할 만한 성능이다. 대표적으로는 연결 리스트의 탐색 연산과 동적 배열에서 배열 마지막에 원소를 추가하는 연산을 제외한 삽입, 삭제 연산이 선형 시간이다.
4. $O(n\log n)$ : 선형 로그 시간이다. 그래프를 그려 보면 $O(n)$과 비교했을 때는 성능이 나쁘지만 $O(n ^ 2)$과 비교하면 성능이 매우 좋다. 대표적인 예로는 비교 정렬 중에서 가장 성능이 좋다고 알려진 퀵 정렬과 병합 정렬 등이 모두 선형 로그 시간이다.
5. $O(n ^ 2), O(n ^ 3)$ : 이제부터는 데이터의 개수가 늘어날수록 연산 횟수가 가파르게 상승한다. $O(n ^ 2)$은 이중 for문을 실행할 때, $O(n ^ 3)$은 삼중 for문을 실행할 때 성능이다.
6. $O(2 ^ n)$ : 지수 시간이다. $O(n ^ 2), O(n ^ 3)$보다 훨씬 성능이 좋지 않다. 데이터 개수가 많을 때 이런 성능을 가진다면 함수 실행이 너무 늦어진다. 정말 어쩔 수 없는 상황이 아니라면 좀 더 나은 알고리즘으로 구현할 수 없을지 고민해 보아야 한다.

### 2.1.3 방심은 금물! : 빅오의 함정
지금까지 빅오를 알아보았는데, 그렇다고 너무 맹신해서도 안된다.
빅오는 연산 횟수를 기반으로 한 상대적인 기준이기 때문에 하드웨어나 다른 외부적인 요소를 전혀 고려하지 않는다.
한 가지 예를 들어보자.
메모리에서 값을 두 개 가져와 더한 후 다시 메모리에 저장하는 동일한 알고리즘을 한 컴퓨터에서는 메인 메모리에서 값을 가져와 저장한다고 가정하자.
컴퓨터 내부에는 클럭(clock)이라는 시계 개념의 장치가 있다.
일단은 단순히 시간이 얼마 걸리는지 잴 수 있다고만 해두자.
메인 메모리에서 값을 가져오거나 저장하는 데는 20 ~ 100 클럭이 필요한 반면, 하드 디스크에서 값을 가져오거나 저장하는 데는 50 ~ 500만 가까운 클럭이 필요하다.
엄청난 차이가 난다.
이런 상황을 고려한다면 동일한 알고리즘을 실행할 때 '둘 다 모두 같은 빅오를 가지므로 비슷한 성능을 내겠구나'라고 판단하는 것은 옳지 못하다.
그러므로 명세에서 빅오가 $O(n)$이라고 표기되어 있더라도 반드시 벤치마킹 등을 이용하여 실제 성능을 테스트해 보는 것이 좋다.

## 2.2 추상 데이터 타입이란
자료 구조를 시작하는데 필요한 마지막 개념으로 추상 데이터 타입(Abstract Data Type, ADT)를 알아보자.
추상 데이터 타입을 이야기하기 전에 먼저 데이터 타입부터 정의해야 한다.

> 데이터 타입(data type)이란 데이터를 저장하는 객체(object)와 객체와 객체가 할 수 있는 연산(operation)의 집합

정의를 보면 두 가지가 눈에 띈다.
객체(object)와 연산(operation)이다.
객체는 데이터 타입이 나타낼 수 있는 모든 값의 집합이다.
컴퓨터는 값을 표현하는데 메모리 제약이 있으므로 정수라고 해도 무한대로 표현할 수는 없다.
그렇다면 표현할 수 있는 최소 정수와 최대 정수가 있을 것이다.
이를 표현하면 {MIN, MIN - 1, ..., -2, -1, 0, 1, 2, ..., MAX - 1, MAX}와 같다.
이 집합이 바로 정수형의 객체인 것이다.

이 책에서 구현할 자료 구조는 대부분 객체 지향적으로 설계되었기 때문에 내부적으로 데이터를 실제로 저장할 객체와 이 객체 위에서 작동할 메서드를 가지고 있다.
연산이란 이 메서드들을 의미한다.
그러므로 연산을 명세할 때는 메서드 이름과 매개변수, 반환값을 반드시 표기해야 한다.

데이터 타입을 알아보았으니 이제는 최종 목적인 ADT를 알아보자.
ADT란 객체와 연산의 명세에서 객체 표현이나 연산 구현을 감춘 것을 의미한다.
이렇게 이야기해서는 이해하기가 쉽지 않다.

예를 하나 들어보자.
C 언어에는 정수형을 표현하는데 32비트 컴퓨터 기준으로 1바이트로 표현하는 char, 2바이트로 표현하는 short, 4바이트로 표현하는 int, 8바이트로 표현하는 long long 등 다양한 데이터 타입이 있다.
int로 선언한 a 변수에 int의 최댓값인 $2 ^ {31} - 1$, 즉 2147483647이 저장된 상태에서 a 변수에 1을 더한다면 정수 오버플로(integer overflow)가 발생하여 프로그램이 의도한대로 작동하지 않는다.
명백한 에러이다.
하지만 파이썬은 자료형을 명시하지 않는다.
a = 2147483647이라고 저장한 후 1을 더해도 잘 더해진다.
정수 오버플로도 발생하지 않는다.

파이썬은 C로 만든 언어이다.
어떻게 이렇게 될 수 있을까?
파이썬은 기본적으로 정수를 int형에 저장한다.
그러다가 최댓값 혹은 최솟값을 넘는 연산이 발생하면 내부적으로 long long으로 변환한다.
프로그래머는 내부적으로 일어나는 일을 신경 쓰지 않아도 된다.
이것이 바로 객체에서 표현이 감추어진 것이다.
포현(representation)이란 어떤 객체를 내부적으로 어떻게 저장할지 고민하는 것이다.
C 언어에서는 직접 메모리 크기를 지정해 주기에 경계 값을 가지지만, 파이썬에는 자료형이 없으므로 실제 메모리에 어떤 크기로 저장되는지 감추어져 있는 것이다.

추상화(abstract)란 개념을 이해하기에 아주 좋은 예이다.
그래서 파이썬을 하이 레벨 언어에서도 추상화가 더 많이 되어 있다고 말한다.
바로 이런 이유 때문이다.
연산도 마찬가지이다.
연산을 명세할 때 함수 이름과 매개변수, 반환값만 표기할 뿐 내부적으로 어떻게 구현(implementation)했는지는 감춘다.

사용자인 프로그래머는 내부 구현과 상관없이 명세만 보고도 이 함수를 잘 사용할 수 있다.
이렇게 추상 데이터 타입을 정의하면 어떤 이점이 있을까?
ADT만 프로그래머에게 제공한다면 데이터 타입 설계자는 내부적으로 객체를 바꾸거나 함수 구현을 변경해야 할 때, 이를 사용자들에게 알릴 필요가 없다.
애초에 프로그래머들은 객체 표현이나 연산 구현을 알고 있지 않기 때문이다.
